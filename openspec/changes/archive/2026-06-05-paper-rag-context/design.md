## Context

当前 `build_paper_context` 将 `paper.full_text[:8000]` 直接放入 System Prompt。论文全文通常 5000-50000 字符，但用户问题只关心某一小节（如 "introduction 讲了什么"）。

改用分块检索后：
- 将全文按段落切分为 chunks
- 用 BM25 + 关键词匹配检索与问题最相关的 3 个 chunks
- 每个 chunk ~800 字符，3 个共 ~2400 字符
- 节省约 5600 字符 ≈ 1400+ tokens（假设 4 chars/token）

## Decisions

### D1: 分块策略

**选择**: 按段落（`\n\n`）切分，每个 chunk 500-1000 字符。小段落合并，大段落再按句子切分。

相比固定长度切分，段落切分保持了语义完整性。

### D2: 检索方式

**选择**: 关键词 BM25 检索（使用已有的 `rank-bm25` 依赖）。

不新建 embedding 向量：
- 论文全文只有一篇，embedding 计算开销大，收益小
- BM25 关键词匹配对中文+英文混合文本效果好
- 无需等待 embedding 生成，即时可用

### D3: 上下文组装

最终注入 LLM 的上下文结构：
```
## 论文信息
标题: ... | 作者: ... | 年份: ...

## 摘要
...

## 相关片段（根据问题 "讲解一下 introduction" 检索）
[片段 1] (相关度: 0.85)
...Introduction 部分内容...

[片段 2] (相关度: 0.72)
...Background 部分内容...

[片段 3] (相关度: 0.68)
...Contributions 部分内容...
```

## Token 对比

| 方式 | 上下文大小 | Token 估算 |
|------|-----------|-----------|
| 之前（全文 8000 字符） | ~8000 chars | ~3500 tokens |
| 之后（3 chunks × 800 字符 + metadata） | ~2800 chars | ~1200 tokens |
| **节省** | **~5200 chars** | **~2300 tokens** |
