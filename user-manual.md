# AstraLoom 用户使用手册

本手册面向两类用户：

- **实验室管理员**：负责部署、配置模型、管理账号、维护论文库和备份。
- **实验室成员**：负责导入论文、阅读讨论、沉淀工具、生成 idea、推进写作和项目协作。

AstraLoom 是实验室自部署系统。每个课题组建议部署自己的实例，组内论文、笔记、API Key、上传文件和研究讨论默认都应留在本实验室控制的服务器或内网环境中。

## 1. 快速开始

### 1.1 管理员部署

推荐先在实验室服务器或个人工作站上用 Docker Compose 启动：

```bash
cp .env.example .env
docker compose up -d
```

启动后访问：

- Web 页面：`http://localhost`
- API 文档：`http://localhost/api/docs`
- 健康检查：`http://localhost/api/health`
- 数据库迁移健康：`http://localhost/api/health/db`

常用检查命令：

```bash
docker compose ps
docker compose exec backend alembic current
docker compose exec backend alembic upgrade head
docker compose logs -f backend
```

生产环境部署前应完成：

- 修改 `.env` 中的 `SECRET_KEY`、数据库密码和模型 API Key。
- 如果服务器不能直连 HuggingFace，设置 `HF_ENDPOINT=https://hf-mirror.com` 后再补论文向量。
- 配置 HTTPS 和访问控制。
- 确认数据库、上传文件、日志和备份不进入 Git 仓库。
- 为至少两位可信成员设置管理员账号，避免唯一管理员不可用。

### 1.2 成员登录

1. 打开管理员提供的 AstraLoom 地址。
2. 注册账号或使用管理员创建的账号登录。
3. 进入系统后先检查设置页中的个人资料、语言和模型配置展示。
4. 如果看不到管理员后台，说明当前账号是普通成员，这是正常状态。

## 2. 界面结构

登录后主要入口包括：

| 入口 | 用途 |
| --- | --- |
| 行动中心 | 查看系统建议的下一步动作，例如补全文、补向量、继续阅读、推进写作 |
| AI 对话 | 通用问答、RAG、网络搜索、文件上传、多模型对话 |
| 项目空间 | 按课题组织论文、方向、写作项目、成员和反馈 issue |
| 论文库 | 搜索、导入、分类、阅读、标记和问答论文 |
| 工具箱 | 沉淀论文中出现的算法、工具、数据集、指标和实验协议 |
| 研究方向 | 生成、评审、讨论和演进 research idea / proposal |
| 写作助手 | 章节级 LaTeX 写作、BibTeX、figures、预览和 AI 辅助写作 |
| 设置 | 个人资料、语言、API 配置展示、Token 用量、论文推送订阅 |
| 管理员 | 用户管理、系统统计、知识库维护和空间治理 |

界面支持全局导航和系统控件的中英文切换。当前业务页面文案以中文为主，后续会逐步补全更完整的英文翻译。

## 3. 论文库

论文库是所有后续功能的基础。建议先积累足够的论文，再进行 idea 生成和写作。

### 3.1 搜索论文

在论文库页面输入关键词后，可以选择：

- **本地搜索**：搜索已经入库的论文。
- **综合学术检索**：聚合外部学术来源，适合发现新论文。
- **arXiv / Semantic Scholar / OpenAlex / Google Scholar**：指定单一来源。

建议优先使用英文关键词，例如 `video grounding`、`multimodal alignment`。如果结果过少，可以放宽年份或换成更宽泛的关键词。

### 3.2 导入论文

导入方式包括：

- 从搜索结果点击导入。
- 输入 arXiv ID 导入。
- 上传 BibTeX。
- 上传 Zotero CSV。

导入后系统会记录导入账号，并在论文库中显示来源用户标签。使用“我的”筛选时，会展示当前账号导入的论文。历史导入数据如果没有导入者，可由迁移或维护脚本统一归到指定账号，例如 `gst`。

### 3.3 重点与有趣标记

论文库支持团队共享标记：

- **重点论文**：组内公认需要认真阅读或作为方向基础的论文。
- **有趣论文**：启发性强、可能有用，但未必是当前方向核心证据的论文。

这些标记对所有成员可见，并可在论文库中筛选。它们不同于个人收藏：

- 个人收藏只影响自己的阅读列表。
- 重点/有趣标记影响团队共同视图。

### 3.4 阅读与划词问答

进入论文详情页后，可以：

- 查看元数据、摘要、标签和分类。
- 打开 PDF 阅读器。
- 选中文本后询问 AI。
- 使用论文问答让模型基于论文全文和检索块回答。
- 添加个人笔记、阅读状态和注释。
- 查看与该论文关联的工具箱条目。

如果 PDF 或全文加载失败，常见原因是论文没有开放 PDF、arXiv 网络不稳定、镜像配置不可用或后台解析任务未完成。可以稍后重试，或让管理员在知识库维护中补全文。

### 3.5 组会报告

在论文库中选择多篇论文后，可以生成组会报告：

1. 勾选论文。
2. 点击组会报告。
3. 在自定义要求框中输入报告侧重点，例如“按方法类别汇总”“重点讨论实验设计”“不要逐篇讲，按研究问题组织”。
4. 生成后导出 Word。

Word 导出会尽量使用中文宋体、英文 Times New Roman，方便继续编辑。

## 4. 工具箱

工具箱用于沉淀“论文里用到的东西”，例如算法、模型、数据集、指标、实验协议和软件工具。

### 4.1 创建工具条目

进入工具箱页面，点击新建后填写：

- 名称。
- 类型：algorithm、method、dataset、metric、protocol、tool 等。
- 摘要。
- 适用场景。
- 局限性。
- 标签。
- 成熟度。

建议把条目写得具体，例如“CLIP-based image-text alignment baseline”比“CLIP”更容易在 idea 生成时使用。

### 4.2 关联来源论文

工具箱条目可以关联论文，并填写：

- 关系：used、baseline、dataset、metric、protocol 等。
- 证据说明：论文中如何使用这个工具、实验结果如何、有什么限制。

论文详情页也可以看到已关联的工具箱条目。

### 4.3 在研究方向中使用工具箱

生成 idea 前，可以在研究方向页面选择工具箱条目，并设置使用模式：

- 参考这些工具。
- 优先使用这些工具。
- 必须使用至少一个选中工具。
- 尽量规避这些工具，寻找替代路线。

这样可以让 proposal 更贴近实验室已有积累，也能避免模型生成完全脱离实际资源的方案。

## 5. 研究方向

研究方向模块用于从论文证据生成和迭代 proposal。

### 5.1 创建方向

建议填写：

- 方向名称。
- 背景描述。
- 关键词。
- 种子论文。
- 可选工具箱条目。

如果论文库中相关论文少于 10 篇，建议先补充论文，否则生成的 proposal 可能证据不足。

### 5.2 生成 Proposal

点击生成后，系统会运行多阶段流程：

1. 生成项目简报。
2. 检索本地论文和外部证据。
3. 生成 Evidence Map。
4. 生成 Gap Map。
5. 多路径生成候选 idea。
6. 去重并保持多样性。
7. 评审并排序 proposal。

再次生成 proposal 不会覆盖原有 proposal，而是新增一批结果和运行记录。你可以比较不同批次的结果。

### 5.3 评审 Proposal

每个 proposal 应重点看：

- 假设是否清楚。
- 与已有工作的差异是否明确。
- 是否引用了足够证据。
- 实验是否可执行。
- 是否需要特定工具、数据或算力。
- 风险是否可控。

评分只是辅助，不建议只按总分选择。更好的做法是把高分 proposal 和有争议但有潜力的 proposal 放到组会讨论。

### 5.4 与 AI 讨论和演进

对一个 idea 可以继续追问：

- 这个方向最薄弱的证据是什么？
- 需要补哪些论文？
- 最小可行实验是什么？
- 哪些 baseline 必须比较？
- 如果实验失败，下一步怎么改？

加入反馈后，可以生成 idea 的新版本。系统会保留谱系关系，方便追踪一个 idea 是如何演进的。

### 5.5 生成实验代码项目

对较成熟的 idea，可以生成代码项目包。理想输出不是单个代码片段，而是项目结构：

- `README.md`
- `requirements.txt`
- `configs/default.yaml`
- `src/data.py`
- `src/model.py`
- `src/train.py`
- `src/evaluate.py`
- `scripts/run_experiment.sh`
- `analysis/plot_results.py`

生成结果仍然需要人工检查，尤其是数据集路径、指标实现、实验假设和 baseline 设置。

## 6. 写作助手

写作助手适合从 proposal 进入论文草稿阶段。

### 6.1 创建写作项目

进入写作助手，创建写作项目后，可以绑定：

- 研究方向。
- 相关论文。
- 项目空间。
- 证据卡。

建议每篇论文或每个投稿目标使用一个独立写作项目。

### 6.2 章节级 LaTeX 写作

写作项目按章节组织。常见章节包括：

- Abstract
- Introduction
- Related Work
- Method
- Experiments
- Conclusion

每个章节都可以输入 LaTeX。预览时系统会合并所有章节，而不是只预览单个章节。

输入 `\` 时，编辑器会提示常用 LaTeX 命令，例如：

- `\section{}`
- `\subsection{}`
- `\cite{}`
- `\ref{}`
- `\begin{figure}`

### 6.3 预览模式

预览可以选择：

- 单栏。
- 双栏。
- 模板样式。

如果模板或 `pdflatex` 缺失，编译会失败。管理员需要在后端镜像或服务器中安装 LaTeX 发行版。

### 6.4 BibTeX 与 Figures

写作项目中会显示：

- **references.bib 面板**：根据已有证据卡生成 BibTeX，可以查看、复制和用于引用。
- **figures 面板**：维护图表清单，并生成 LaTeX figure 插入片段。

文件结构中的 `references.bib` 和 `figures/` 是快捷入口，点击后会定位到对应面板。

### 6.5 AI 辅助写作

每个章节都可以唤出 AI 辅助：

- 扩写段落。
- 改写为学术表达。
- 检查逻辑跳跃。
- 推荐引用。
- 根据证据卡生成段落。
- 对 claim 做风险检查。

AI 输出需要人工复核。尤其是引用、实验结果、数字和贡献描述，不应直接无检查地放入论文。

## 7. 项目空间

项目空间适合按“一个课题 / 一篇论文 / 一个方向”组织。

### 7.1 创建空间

创建空间时建议填写：

- 空间名称。
- 研究目标。
- 当前阶段。
- 成员分工。

### 7.2 绑定资源

空间可以绑定：

- 论文。
- 研究方向。
- 写作项目。

绑定后，空间详情页会展示资源列表、进度看板、下一步建议和活动记录。

### 7.3 成员角色

- owner：空间所有者，可管理成员和资源。
- editor：可编辑和绑定资源。
- viewer：只读查看。

### 7.4 反馈 Issue

项目空间支持类似 GitHub issue 的反馈入口。适合记录：

- 论文阅读问题。
- 实验待办。
- 写作修改建议。
- 组会反馈。
- 数据或代码风险。

## 8. AI 对话

AI 对话适合做快速问答、搜索增强和文件分析。

### 8.1 模型选择

设置页中可以查看当前模型配置。系统支持：

- DeepSeek。
- OpenAI-compatible API 端点。

如果兼容端点支持 OpenAI Responses API，部分模型可以返回 reasoning summary。是否能显示思考过程取决于模型、端点和系统配置。

### 8.2 RAG 与网络搜索

- 开启 RAG：模型会参考本地论文库。
- 开启网络搜索：模型会参考实时网页结果。
- 同时开启：适合需要结合组内论文和外部信息的问题。

### 8.3 文件上传

支持上传：

- 图片：仅对支持多模态的 GPT 兼容模型有效。
- PDF：系统提取文本后加入上下文。
- 文本/Markdown：直接加入上下文。

如果使用 DeepSeek 且当前模型不支持图片，图片不会作为视觉内容被模型理解。

## 9. 设置与管理员维护

### 9.1 个人设置

可修改：

- 显示名称。
- 邮箱。
- 密码。
- 头像。
- 界面语言。

### 9.2 Token 用量

设置页可以查看：

- 个人 Token 用量。
- 调用历史。
- 费用估算。
- 管理员全局统计。

费用估算会按模型区分，不同模型的输入、输出价格可以不同。

### 9.3 论文推送

论文推送基于关键词订阅：

1. 在设置中填写关键词。
2. 设置推送时间。
3. 保存并测试。
4. 在推送中心查看每日推荐。
5. 对论文标记感兴趣、稍后、不感兴趣，以改善后续排序。

### 9.4 知识库维护

管理员可以维护：

- 缺失全文。
- 缺失向量。
- 检索质量诊断。
- 用户和空间治理。

建议定期检查迁移状态和后台任务日志。

如果补向量失败并且后端日志出现 `huggingface.co`、`all-MiniLM-L6-v2` 或 `Network is unreachable`，说明本地向量模型还没有下载成功。管理员可以在服务器 `.env` 中加入：

```bash
HF_ENDPOINT=https://hf-mirror.com
EMBEDDING_MODEL_NAME=all-MiniLM-L6-v2
```

然后重建或重启 backend / celery-worker，再重新执行补向量。模型会缓存到 Docker volume `model_cache`，后续不会每次重新下载。

## 10. 推荐使用流程

### 10.1 新课题启动

1. 创建项目空间。
2. 搜索并导入 10-30 篇核心论文。
3. 标记重点论文和有趣论文。
4. 从论文中沉淀工具箱条目。
5. 创建研究方向并选择种子论文和工具。
6. 生成 proposal。
7. 组织组会讨论并演进 idea。
8. 生成实验计划和代码项目。
9. 创建写作项目，开始章节级 LaTeX 写作。

### 10.2 日常论文阅读

1. 查看论文推送或搜索新论文。
2. 导入有价值论文。
3. 设为待读或重点/有趣。
4. 阅读 PDF 并划词问答。
5. 将重要算法、数据集或协议加入工具箱。
6. 把论文绑定到对应项目空间。

### 10.3 写作阶段

1. 从成熟 idea 创建或绑定写作项目。
2. 准备证据卡和 BibTeX。
3. 先写 Introduction / Method 的骨架。
4. 使用 AI 辅助局部段落，而不是一次性生成整篇。
5. 合并预览全文。
6. 检查引用和 claim。
7. 导出 LaTeX / Word 或继续在 Overleaf 等工具中排版。

## 11. 常见问题

### Q: 搜索论文时报数据库字段不存在？

通常是代码已经更新，但数据库迁移未执行。管理员应运行：

```bash
docker compose exec backend alembic upgrade head
docker compose restart backend celery-worker celery-beat
```

### Q: LaTeX 不能编译？

常见原因：

- 后端环境没有安装 `pdflatex`。
- 模板依赖的 class 或 package 缺失。
- LaTeX 源码本身有语法错误。

可以先切换单栏/双栏基础预览，确认源码无误后再使用模板。

### Q: GPT 回复比 DeepSeek 慢？

可能原因包括：

- 兼容端点网络延迟更高。
- reasoning 模型消耗更多推理时间。
- 输出 token 上限更大。
- 端点没有稳定流式返回。
- 当前请求包含图片、全文或较长上下文。

建议先用测试脚本验证端点延迟，再调整模型、输出预算和搜索/RAG 开关。

### Q: Idea 生成超时？

Idea 生成会检索论文、构建证据、调用多次模型并做评审。论文很多、外部搜索慢、模型响应慢或一次生成数量过大都会超时。建议：

- 降低单次生成数量。
- 优先使用本地论文库。
- 减少外部搜索依赖。
- 分批生成并比较，不要一次要求过多 proposal。

### Q: 哪些文件不能上传到 GitHub？

不要提交：

- `.env`
- API Key
- 论文 PDF
- 用户上传文件
- 数据库数据和备份
- 私有笔记
- 日志
- Token 和真实用户信息

可以提交源码、OpenSpec、README、用户手册、示例配置和测试。

## 12. 维护建议

- 每次功能开发前先写 OpenSpec。
- 每次完成变更后提交 git 并归档 OpenSpec。
- 定期运行数据库迁移检查。
- 定期备份 PostgreSQL 和上传文件目录。
- 定期清理无用日志和临时文件。
- 对论文和未公开研究内容保持最小权限访问。
- 对 AI 生成内容保持人工复核，尤其是引用、实验结果和结论。
